# Skill: Markdown to DOCX Converter

## Purpose

Convert markdown files (PRDs, PRFAQs, field notes) generated by this system into properly formatted .docx files using docx-js. The output should match [COMPANY] document conventions and be ready to paste into Hermes or share with stakeholders.

---

## Routing

**Type**: Standalone utility skill, separate from pipeline routing

**Invocation**: Direct user request with conversion keywords

**Trigger phrases:**
- "Convert [filepath] to docx"
- "Export [filepath] as Word doc"
- "Turn the compliance benchmarking PRD into a docx"
- "Make a Word version of [filepath]"

This skill is NOT part of the ingestion or pipeline workflows. It operates on already-saved output artifacts.

### Required
- **Markdown file path**: Typically from `/context/product-context/prds/`, `/context/product-context/prfaq/`, `/context/customer-notes/`, or `/context/insights/`

### Optional
- **Output filename override**: Default is same name as input with `.docx` extension

## Markdown → DOCX Mapping

Apply these mappings exactly — do not improvise alternatives.

### Typography

- **Default font**: Arial, 12pt
- **Page size**: US Letter (12240 x 15840 DXA) — CRITICAL: docx-js defaults to A4, always override explicitly to US Letter
- **Margins**: 1 inch all sides (1440 DXA)
- **Orientation**: Portrait

### Headings

| Markdown | Style | Font Size | Weight | Color | Outline Level |
|----------|-------|-----------|--------|-------|---------------|
| # (H1) | Heading1 | 16pt | Bold | #2E4057 | 0 |
| ## (H2) | Heading2 | 14pt | Bold | #2E4057 | 1 |
| ### (H3) | Heading3 | 12pt | Bold | #666666 | 2 |

All headings must include `outlineLevel` for TOC compatibility.

### Body Text

- **Regular paragraphs**: Normal style, Arial 12pt
- **Paragraph spacing**: 120 DXA after each paragraph
- **Bold** (`**bold**` or `__bold__`): TextRun with `bold: true`
- **Italic** (`*italic*` or `_italic_`): TextRun with `italics: true`
- **Inline code** (`` `code` ``): TextRun with `font: "Courier New"`, `size: 20` (10pt in half-points)

### Lists

**CRITICAL**: NEVER use unicode bullet characters (•, ◦, etc.). Always use docx-js numbering system.

#### Unordered Lists (`-`, `*`)

```javascript
{
  reference: "unique-bullet-list-id",  // MUST be unique per list
  levels: [{
    level: 0,
    format: LevelFormat.BULLET,
    text: "•",
    alignment: AlignmentType.LEFT,
    style: {
      paragraph: {
        indent: { left: 720, hanging: 360 }
      }
    }
  }]
}
```

#### Ordered Lists (`1.`)

```javascript
{
  reference: "unique-number-list-id",  // MUST be unique per list
  levels: [{
    level: 0,
    format: LevelFormat.DECIMAL,
    text: "%1.",
    alignment: AlignmentType.LEFT,
    style: {
      paragraph: {
        indent: { left: 720, hanging: 360 }
      }
    }
  }]
}
```

#### Nested Lists

- Increment `level` for each nesting depth (0, 1, 2)
- Adjust indent: `left: 720 * (level + 1)`, `hanging: 360`
- Each list reference must be unique to avoid numbering continuation errors

### Tables

**CRITICAL**: Use `WidthType.DXA` (never `PERCENTAGE` — breaks in Google Docs)

```javascript
{
  width: {
    size: 9360,           // Content width (US Letter - margins)
    type: WidthType.DXA
  },
  columnWidths: [3120, 3120, 3120],  // Must sum to 9360
  rows: [
    new TableRow({
      children: [
        new TableCell({
          width: { size: 3120, type: WidthType.DXA },  // Dual width required
          shading: {
            type: ShadingType.CLEAR,
            fill: "2E4057"  // Header: dark
          },
          margins: { top: 80, bottom: 80, left: 120, right: 120 },
          children: [
            new Paragraph({
              children: [
                new TextRun({ text: "Header", bold: true, color: "FFFFFF" })
              ]
            })
          ]
        })
      ]
    })
  ],
  borders: {
    top: { style: BorderStyle.SINGLE, size: 1, color: "CCCCCC" },
    bottom: { style: BorderStyle.SINGLE, size: 1, color: "CCCCCC" },
    left: { style: BorderStyle.SINGLE, size: 1, color: "CCCCCC" },
    right: { style: BorderStyle.SINGLE, size: 1, color: "CCCCCC" },
    insideHorizontal: { style: BorderStyle.SINGLE, size: 1, color: "CCCCCC" },
    insideVertical: { style: BorderStyle.SINGLE, size: 1, color: "CCCCCC" }
  }
}
```

**Table styling:**
- Header row: `fill: "2E4057"`, text white, bold
- Alternating rows: `fill: "F5F5F5"` and `fill: "FFFFFF"`
- Border color: `"CCCCCC"`
- Cell margins: `top: 80, bottom: 80, left: 120, right: 120`
- Use `ShadingType.CLEAR` (never `SOLID` — causes black backgrounds)

### Code Blocks (``` ```)

Render as a shaded paragraph:

```javascript
new Paragraph({
  text: codeContent,
  style: "Code",
  shading: {
    type: ShadingType.CLEAR,
    fill: "F4F4F4"
  },
  indent: {
    left: 720  // Indent for visual distinction
  },
  font: "Courier New",
  size: 20  // 10pt in half-points
})
```

### Horizontal Rules (`---`)

Render as a paragraph with bottom border:

```javascript
new Paragraph({
  text: "",
  border: {
    bottom: {
      style: BorderStyle.SINGLE,
      size: 6,
      color: "CCCCCC"
    }
  }
})
```

Do NOT use tables as dividers.

### Blockquotes (`> text`)

```javascript
new Paragraph({
  children: [
    new TextRun({ text: quoteText, italics: true })
  ],
  indent: {
    left: 720  // Left indent
  },
  border: {
    left: {
      style: BorderStyle.SINGLE,
      size: 12,
      color: "CCCCCC"
    }
  }
})
```

### YAML Frontmatter

Strip entirely — do not render in the docx. Frontmatter is delimited by `---` at the start and end of the block at the top of the file.

## [COMPANY] Document Conventions

Apply these on top of the base markdown mapping:

### Document Header

1. **Title** (from first H1): Centered, Arial 18pt, bold, color #2E4057
2. **Confidentiality notice**: "[COMPANY] Confidential", Arial 10pt, centered, color #999999
3. **Horizontal rule**: Below the header block

```javascript
new Paragraph({
  children: [
    new TextRun({ text: documentTitle, bold: true, size: 36, color: "2E4057" })
  ],
  alignment: AlignmentType.CENTER
}),
new Paragraph({
  children: [
    new TextRun({ text: "[COMPANY] Confidential", size: 20, color: "999999" })
  ],
  alignment: AlignmentType.CENTER,
  spacing: { after: 240 }
}),
new Paragraph({
  text: "",
  border: {
    bottom: { style: BorderStyle.SINGLE, size: 6, color: "CCCCCC" }
  },
  spacing: { after: 240 }
})
```

### Document Footer

Use footer sections with tab stops (not tables):

```javascript
{
  default: new Footer({
    children: [
      new Paragraph({
        children: [
          new TextRun({ text: "[COMPANY] Internal", size: 18, color: "999999" }),
          new TextRun({ text: "\tPage ", size: 18, color: "999999" }),
          new TextRun({ children: [PageNumber.CURRENT], size: 18, color: "999999" })
        ],
        tabStops: [
          { type: TabStopType.RIGHT, position: TabStopPosition.MAX }
        ]
      })
    ]
  })
}
```

## Implementation Instructions

### Step 1: Setup

Install docx-js globally:
```bash
npm install -g docx
```

Create output directory if needed:
```bash
mkdir -p /context/outputs
```

### Step 2: Parse Markdown

Do NOT use a markdown parsing library. Parse directly using string operations and regex to avoid dependency issues.

**Parsing order:**
1. Strip YAML frontmatter (between `---` delimiters at file start)
2. Extract first H1 as document title
3. Split remaining content by section markers (`#`, `##`, `###`)
4. Parse each section for inline formatting, lists, tables, code blocks, blockquotes
5. Build docx-js structure incrementally

**Key regex patterns:**
- Headings: `/^(#{1,3})\s+(.+)$/gm`
- Bold: `/\*\*(.+?)\*\*/g` or `/__(.+?)__/g`
- Italic: `/\*(.+?)\*/g` or `/_(.+?)_/g`
- Inline code: `/`(.+?)`/g`
- Lists: `/^(\s*)([-*]|\d+\.)\s+(.+)$/gm`
- Tables: `/^\|(.+)\|$/gm`
- Code blocks: `/```(.+?)```/gs`
- Blockquotes: `/^>\s+(.+)$/gm`
- Horizontal rules: `/^---$/gm`

### Step 3: Build docx-js Document

Create Document with sections:

```javascript
const doc = new Document({
  sections: [{
    properties: {
      page: {
        width: 12240,   // US Letter width
        height: 15840,  // US Letter height
        margin: {
          top: 1440, right: 1440, bottom: 1440, left: 1440
        }
      }
    },
    headers: {
      default: new Header({ children: [/* header content */] })
    },
    footers: {
      default: new Footer({ children: [/* footer content */] })
    },
    children: [
      /* Document content paragraphs, tables, etc. */
    ]
  }]
});
```

### Step 4: Generate File

```javascript
Packer.toBuffer(doc).then(buffer => {
  fs.writeFileSync(outputPath, buffer);
});
```

### Step 5: Validate

Always run validation after creating the file:

```bash
python scripts/office/validate.py /context/outputs/{filename}.docx
```

If validation fails:
1. Unpack the docx (it's a zip file): `unzip {filename}.docx -d temp/`
2. Fix XML issues in `temp/word/document.xml`
3. Repack: `cd temp && zip -r ../{filename}.docx * && cd ..`
4. Re-run validation

**Common validation failures:**
- Missing required XML namespaces in document.xml
- Invalid shading values (use CLEAR, not SOLID)
- Width type mismatches (use DXA everywhere)
- Duplicate numbering IDs (ensure unique list references)

## Error Handling

### Input Validation Errors

| Error | Message | Action |
|-------|---------|--------|
| File not found | "File not found: [path]" | Ask user to confirm path |
| Wrong extension | "Input must be a .md file" | Ask user to provide markdown file |
| Empty file | "File is empty" | Ask user to confirm file content |

### Conversion Errors

| Error | Fallback Behavior |
|-------|-------------------|
| Malformed table | Skip table, log warning |
| Invalid heading level | Treat as H3 |
| Unclosed code block | Treat rest of document as code |
| Unclosed blockquote | Treat rest of section as blockquote |

Always complete the conversion even if some elements fail. Log warnings for skipped elements.

## Notes

- **US Letter vs A4**: docx-js defaults to A4. ALWAYS override page size to US Letter (12240 x 15840 DXA) explicitly.
- **DXA vs PERCENTAGE**: ALWAYS use `WidthType.DXA` for tables. PERCENTAGE breaks in Google Docs and other non-Word viewers.
- **Unique list IDs**: Each list must have a unique numbering reference. Reusing IDs causes numbering to continue incorrectly.
- **Shading type**: Use `ShadingType.CLEAR`, never `SOLID`. SOLID renders as black backgrounds in some viewers.
- **Dual width declaration**: Tables require width set on BOTH the table and each cell. Setting only one causes rendering issues.
